Special Comments
SC1) The field final indicates whether the transaction can be updated by non admin users.  A true value means it cannot be updated by non admin users.
SC2) The field fund must match (case sensitive) an existing fund name in eTapestry.  Existing fund names can be found at Manage...Funds.
SC3) The field campaign must match (case sensitive) an existing campaign name in eTapestry.  Existing campaign names can be found at Manage...Campaigns.
SC4) The field approach must match (case sensitive) an existing approach name in eTapestry.  Existing approach names can be found at Manage...Approaches.
SC5) The field letter must match (case sensitive) an existing letter name in eTapestry.  Existing letter names can be found at Manage...Letters.
SC6) Only one of these fields should be populated.  If an add is being performed and no schedule is provided, a default schedule (the frequency will be annual, the first installment date will be the date of the pledge, and the installment amount will be the initialPaymentAmount or the pledge amount) will be created.  If an update is being performed on a transaction that has a schedule, a schedule will be required.  On an update, switching schedule types (standard versus custom) is allowed.
SC7) The field paymentValuable should only be populated when initialPaymentAmount is above zero.  If an update is being performed on a pledge that already has an initial payment amount, paymentValuable will be required.
SC8) The field scheduledValuable is required when standardSchedule or customSchedule is marked with a process type of auto.  Auto transactions must have CreditCard or ElectronicFundsTransfer populated.
SC9) The field softCredit contains the soft credit transaction linked to this pledge.  The linked soft credit can have a different fund, campaign, approach, letter, and amount.  Please note, soft credit transactions can only belong to donor role accounts.  If an update is processed, the soft credit account can be switched from one account to another.
SC10) The field tributeAccountRef indicates the transaction is in honor/memorial of a particular account.  This must be a tribute role account.
SC11) The field recognitionName should only be populated when recognitionType is 3.
SC12) The field writeOff indicates the remaining balance of the pledge has been written off (i.e. the donor has notified the organization they no longer plan to fulfill the pledge or the organization simpily doesn't expect the donor to make any more payments).
SC13) eTapestry allows transactions to be reversed.  The field offsettingRef indicates this transaction is a transaction that has been reversed or the reversal transaction.  Reversal transanctions or transactions that have been reveresed cannot be updated.
SC14) eTapestry allows segmented pledge transactions which consist of multiple pledges.  The field segmentedTransactionRef indicates this pledge is a single segment of a segmented pledge transaction.  Individual segments of a segmented pledge transaction cannot be updated, but the entire segmented pledge transaction (which includes the individual segments) can be updated.
SC15) eTapestry allows transactions to denote a tribute (tributes are separate accounts within the database).  For reporting purposes, eTapestry stores a replica of the transaction within the tribute's journal.  When the field originalAccountRef, originalAccountName, and originalTransactionRef are populated, this transaction is a tribute's replica.  The field originalAccountRef will contain the account ref of the original transaction owner.  The field originalAccountName will contain the account name of the original transaction owner.  The field originalTransactionRef will contain the ref of the original transaction.  Please note, replica tribute transactions cannot be updated.  If attempted, soap fault 107 will occur.
SC16) Journal entries retrieved from eTapestry will contain all attachment data except the fileContent.  This is done for efficiency purposes.
